## DMM Crypto (2023/02 ~ 2024/12) ※DMM BITCOIN とは別の会社

ブロックチェーン構想と、ゲームプラットフォームが実現できそうということで参画

### Seamoon Protocol

Seamoon Protocol というブロックチェーン構想の開発に参加しました

- OASYS L2 に DM2 Verse ブロックチェーンをデプロイ。ガス代無料で高速処理ができ、ゲームプレイに最適
- SMP トークン(ERC20)を発行
- 既存IPや新規ゲーム、3rdパーティも誘致しweb3ゲームを多くリリース
- 情報やユーザの滞留場所としてのポータルサイト

多くのゲームやデベロッパーを集約

Seamoon Protocol 構想が解散したことによりチームを離れました

[公式サイト](https://seamoon.dmm.com/)

[ホワイトペーパー](https://docs.seamoon.dmm.com/whitepaper)

**技術スタック**

- 言語、フレームワーク、DB、インフラ、SaaS
- ブロックチェーン部分（あまり多くはタッチしていません）
    - Ethereum, Solidity, Web3Auth
- バックエンド部分
    - TypeScript, NestJS, Next.js, PostgreSQL, AWS, Auth0
    - 自分がアーキテクトとして選定しました
- ゲーム部分
    - Unity, C#
- アーキテクトとして
    - DDD, RDRA

**チーム状況**

- エンジニア人数
    - ジョイン時4人、最終的に約15人
- 自分のロール

    - CTOのサポートする形でカルチャー醸成的なことも少し

**業務の詳細**

- アーキテクトとして
    - プロダクト開発の優先度について判断
        - チームのミッションに合致するかどうかなど判断軸に沿った提案
        - サードパーティに役立つプロダクトを多く作る計画があったが、当時の開発キャパシティ的に選択と集中が必要と感じ、会社のビジョンやホワイトペーパーに合致するプロダクトを選定して優先的に開発することをCTOに提案。たとえサードパーティに便利でも、自分たち（プラットフォーマー）の作りたい世界観のメッセージが誤って伝わる可能性があるプロダクトは除外を提案など、CTOと壁打ちし多く意見が進みました
    - 独自のプロダクト関連図を作成し、Seamoonというプラットフォーム全体が構築するシステム、提供するサービスの全体像を作成しイメージを共有しました
        - 提供する機能群と、2種類（ユーザ向け、サードパーティ開発者向け）のフロントエンド、APIの形で提供する機能群、そのAPIを利用するUnityクライアント、などなど
        - blockchainへの操作は、直接rpcエンドポイントを叩く方法とAPIが用意されていて、3rdはどちらも利用可能で、その区別としては…
        - プロダクトやデプロイ単位を図解してまとめ、プラットフォーム全体としてどのような構成になるのか、どこから開発を始めるべきかという考慮のベースをCTOと共有しました。それは開発スケジュールやチーム構成、採用戦略に使われました
    - 仕様策定に際し、[RDRA](https://www.rdra.jp/)を記述（主にユースケース複合図＋業務フロー図）しながら、必要なAPIとストレージ（永続化層）について定めていくという作業を必要に応じて行なっていました
    - 主要技術スタックの選定を行いました
    - 言語：TypeScript
        - プロダクトのコアとなる Ethereum に関するライブラリが頻繁に更新されていて信頼できるものがあること
        - 型があり、インターフェイスプログラミングができる
            - プロダクト寿命が長く、拡張を繰り返すことが想定されるため、保守性を維持できるかという観点
        - モジュール性があり、疎結合に保った実装が可能か
            - コードベースが大きくなっても秩序を保つため。**カプセル化**ができること
            - 将来的にサービスを分離（マイクロサービス化）することになった時、機能単位で疎結合になっていれば容易にマイクロサービス化が可能になる
                - 「**境界づけられたコンテキスト**」による設計を、コード（モジュール）で表現するのがモノリス、サービスで表現するのがマイクロサービスという違いだけだという考えに至りました
        - バックエンド・フロントエンドともに信頼の高いフレームワークが利用できること
        - テストフレームワークがあるか
        - 即戦力エンジニアの採用しやすさ
    - バックエンドフレームワーク: NestJS
        - バックエンド専用として作られたTypeScriptのフレームワークであり、moduleを利用したDI、カプセル化が可能で、将来的にサービスが拡大してマイクロサービス化することをある程度想定して設計できる（後述）ところに魅力を感じ採用
    - フロントエンドフレームワーク：Next.js
        - TypeScriptのデファクトと言えるフレームワークの1つ
    - マイクロサービスか、モノリスか
        - マイクロサービスをやりたいメンバーが多かったですが、エンジニアの人数（5~6人）、現状のプロダクト規模を考え、モノリスを基本としました。
        - ただし前述の通り、プロダクトの構想が進めば（部分的にも）マイクロサービス化が適切になるケースは想定できたため、コードは疎結合に作る指針で進めました。これは境界づけられたコンテキストの考えでいくと極めて自然でした。**モノリスであろうがマイクロサービスであろうが、ドメインモデリングは変わらない**ということを見出しました
        - コーディングスタイルやER図を作成しチームに共有しました
        - ただし、特定の1サービスだけ先行リリースが求められたのと、担当開発メンバーの得意分野も考慮し、そのサービスだけマイクロサービス化して切り出し、技術スタックも上記とは異なる開発を許容するなど柔軟に対応しました
            - これもこれで開発は順調に進み、また互いのバックエンドサービス同士もかなり疎結合を保ったまま開発が進めることができ、フロントエンドも適切にサービスを呼び分けることができていました。かつ、デプロイ頻度や品質管理の社内フローも結構異なっていたので、むしろマイクロサービス化して正解だったと感じています
- Unity・ウォレット
    - 上記で実装したような公開APIをUnityゲームから利用しやすくするため、Unityクライアントを実装しました
    - コーディングスタイルの統一、ライブラリやアセットの導入判断（UniRx,UniTaskはデファクトとみなし導入、Odinは有料だから無し,etc）
    - クレデンシャルな設定(client_id,client_secretとか)はプレハブ、シーン配置オブジェクトにインスペクタ上で設定するだけみたいな感じに
    - サンドボックス環境も整備（バックエンド側の話）
    - Unityクライアント経由のAPIにあえて制限をかけて使いやすくするなどの設計も

        - デベロッパーの負担ができるだけ少ないような設計でラップしていました
    - Unityとブロックチェーンウォレットの両方の知識が求められる部分であり、チームにとっても重要なプロダクトのひとつということでやっていました
- CMS導入、ドキュメンテーションとか
    - データ入稿用にヘッドレスCMSを導入
    - エンジニアの手離れが早くできることが重要と考えているため、ドキュメントも重視しています
    - プランナーが理解できるように利用手順をNotionに記載
        - 意識して書くようにしている主な内容

        - 共有ページの管理方法などを策定、助言などしました

### デッドストームパイレーツ(2008/10 ~ 2010/4)

- アーケード用ゲーム
- 2人乗りライド型筐体のガンシューティング
- [https://www.youtube.com/watch?v=XTF9yzWm8oE](https://www.youtube.com/watch?v=XTF9yzWm8oE)

**技術スタック**
- C++、PS3
- PS3互換基盤のSYSTEM357。後にPS3に移植も担当しました

**チーム状況**
- エンジニア10~20人

**業務の詳細**

<details><summary>クリックして開く</summary>

- 最初はリグを実装しました
    - キャラクターの髪や服の裾が、風や重力、キャラのモーションによって自然に揺れる表現です
    - ボーン（骨）の一本一本にかかる力や慣性力を計算し、隣の骨との影響を計算し、自分の体にめり込まないよう当たり判定の計算と押し出し処理を計算し…というのを3D計算しました
    - 骨の設定や、当たり判定の登録などをデザイナが Maya で登録できるようにし、データの出力とランタイム読み込み実行を行う処理も行いました。（python で記述する Maya ツールの部分などテクニカルアーティストの人と協力）
    - それまで社内で格ゲーなどで使われたプログラムがあって渡されたのですが、最終的にほとんど自分で一から書き直しています
    - この時実装したプログラムはのちに社内の共通ライブラリに取り入れられました
    - 3D物理計算を行い、行列計算を実行するという、かなり物理や数学が求められる部分で、難度は高かったものです
    - キャリアの現時点においても、プログラミングのタスクとしてこれより難度が高い仕事には未だ出会っていません
- レース部分
    - 自機である海賊船を、舵輪型コントローラで制御し動かすミニゲームがあるのですが、その処理全般を実装しました
    - 船が舵を切った時の自然な動きと、ゲームとしてわかりやすく気持ちいいケレン味のある動きとのちょうど良いところを模索しました
    - 岸壁に衝突して強制的にコースに戻すなどの処理や、岸壁データの作成、読み込み周りを丸ごと作りました
    - ほぼレースゲームを作るのと同じ仕事だ、と先輩に言われました
- UI
    - ゲーム中のさまざまなUI表示を実装
- クレジット
    - 100円玉を入れるとプレイ回数が1回分増えるといった処理のことです
    - ゲームオーバーでコンティニューしている最中のぎりぎりのタイミングで100円玉を入れたらどう解釈するかとか、（さらに2人プレイなのでどちらのクレジットとみなすか）などお金が直接絡むこともあり見た目以上にナーバスな部分。決済系APIを使うときと同じようなシリアスさがあります
    - アーケードゲーム特有の問題点をハードウェア面も考慮しながら要件定義と実装しました
        - 安全性の面です。筐体（ゲーム機）が揺れるゲームなので、プレイヤーがお金を入れ（ようとしていると思われ）る時は揺れを止めるなどの制御も同時に行う必要があります
    - この時書いたのはフローチャート程度でしたが、設計をしっかり固めてから実装に入るということを最初に意識した仕事でもあります
- 立体視対応
    - 当時立体視が流行っていたこともあり、ゲームの立体視対応を行いました
    - 立体視対応をやってくれるライブラリのようなものが存在しない頃ですので、射影変換行列を、右目用と左目用に変換する計算を自力で行い行列計算を行いました。（剪断という言葉をこの時初めて知りました）
        - オブジェクトの座標をカメラ視点の座標系に変換する処理のことです
        - 立体視は人間の両目の視差の分（だいたい60~70mm）だけずらすのですが、これはスクリーンのサイズに関わらず固定となるため（つまり画面に直接ものさしを当てて60mmずれるように映す必要がある）、テレビに映すか巨大なスクリーンに投影するかで射影変換行列のパラメータが変わります
- モーションコントローラ対応
    - PS3移植するとき、モーションコントローラ対応での発売だっため、その辺の対応もしました
- 全体的に、難度の高い課題が多かったようです
    - 新卒で入った当時はそれが普通だと思ってあまり気にしていませんでしたが、その後のキャリアでこのころより難しい課題にほとんど遭遇しないので、むしろそういうハイレベルな課題を新卒の自分を信じて与えて続けてくれた当時のマネージャに感謝しています
</details>
